# Changelog

All notable changes to the **publicly released `dmtool` artifacts** — the native CLI binary and the agent plugins (Claude Code, OpenAI Codex). This file is maintained in the source repo and shipped to the public [`a12-dmtool-releases`](https://github.com/mbackschat/a12-dmtool-releases) mirror; it tracks only what an end user can download, not internal development.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versioning is [SemVer](https://semver.org/) plus **A12 Kernel compatibility metadata** (the kernel each release targets is recorded per entry, never folded into the version string).

## [0.5.1] — kernel 30.8.1 (A12 Tools 2025.06-ext5)

### Fixed

- **Installing the plugin mid-session no longer leaves the agent guessing how to get `dmtool`.** The binary is delivered by a `SessionStart` hook, which does not fire at `/plugin install` — so `dmtool` isn't on `PATH` in the session you installed it in, and the agent would try to download or build it on its own. The bundled skill now tells the agent, on a missing `dmtool`, to point you at a **context-preserving resume** (`claude --continue` / `--resume` — it keeps your whole conversation, no `/clear` needed) and to **not** self-install the tool; the README's activation note emphasizes the same. The CLI binary itself is unchanged from 0.5.0.

## [0.5.0] — kernel 30.8.1 (A12 Tools 2025.06-ext5)

A document-model **editor-coverage** wave: the structure/metadata edits the A12 Tools editor exposes are now CLI verbs (so an agent never hand-edits the model JSON), plus self-describing, corrective diagnostics for rule/computation placement.

### Added

- **Change a group's repeatability in place — `group modify --repeatable <max>` / `--non-repeatable`.** Repeatability could be set only at `group add`; making an existing group a repetition list (or collapsing one) meant hand-editing the JSON. Re-bound to the kernel gate, so a change that breaks the model fails loudly.
- **Set a group's repetition row-key — `group modify --index-field <fieldName>` / `--clear-index-field`.** Designates a direct-child field as the row-key (requires a repeatable group); `group read` echoes it.
- **Register / remove a display locale — `config modify --add-locale <code>` / `--remove-locale <code>`.** Adding a locale then requires every rule/computation to carry its message (provision those first, or do both in one `apply`); removing refuses the last locale.
- **Manage the model's access-control roles — `config modify --role <name>` (merges) / `--clear-roles`.** A merge-safe peer of the raw `--annotation roles=…`; `config read` surfaces the header annotations so it round-trips.
- **Set a field/group display label — `meta <ref> --label <locale>=<text>`.** Merges per locale; the `meta` read echoes the label.
- **Rich type definitions — `typedef add --id <id> <spec.json>`** for an ENUM (`enum.values`) or a restricted STRING (`string.pattern`), the same per-kind spec a `field add` takes; and **`typedef unimport --reference <r>`** removes a type-definition import (the inverse of `typedef import`, re-validated so a still-used import is refused).
- **Rename a model's id — `model rename --to <newId>`, safety-gated.** With a workspace (`-w`) it refuses when a sibling includes/imports the old id (`RK_REFERENCED`) or the new id collides (`RK_NAME_EXISTS`); nothing is written.
- **Field-level custom error messages — `requiredMessages` + `enum.errorMessages`** on the `field add`/`field modify` spec (and the `apply` field ops): per-locale custom text for a required field left empty / an invalid enum value. Omit to use the kernel default.
- **Enum-category `description`** — an editor-level note on an enum category that the kernel tolerates.
- **`rule add` spec key `allowDifferingDecimals`** — the editor's "Allow Differing Decimal Places"; prepends `@SuppressWarning(MVK_INVALID_COMPARE_DEC_PLACES)` so a field-vs-field `==`/`!=` across differing decimal scales is accepted (the kernel's one suppressible warning).
- **`batch -m <model>` is the default model for ops that omit their own** — an op with its own `-m` still wins.
- **`field read` / `field remove` accept `--group`/`--name`, not just a positional path** — the same locator style as `field add`.

### Changed

- **`field read` echoes the field's `required` state** (`true` / `false` / `"ifParentPresent"`), so a requiredness edit is verifiable straight from the tool.
- **`rule modify --message` merges per-locale** instead of replacing the whole message set (adding one locale's message no longer silently drops the others).
- **`typedef read` surfaces import provenance** — `data.importedFrom` distinguishes local from imported type definitions (resolve to files via `model info`).
- **More model conventions are discoverable from `--help`** — `config read` explains the locale registry + per-locale-message rule; `workspace roles` the roles/users/annotation contract; `apply`/`batch --help` point at `schema apply`/`schema batch`.
- **The internal/external description wording is sharpened** on the `field` schema and the `meta --internal`/`--external` flags: *external* = the end-user-facing hint shown in the form UI; *internal* = a maintenance note shown read-only to Form modelers, never to end users.
- **`apply` covers the widened verbs** — the new typedef/group/config edit ops run inside an `apply` transaction at parity with their standalone forms (single-sourced, so they can't drift).

### Fixed

- **INFO rule severity is now advertised on the `rule add` / `rule check` surface.** It worked end-to-end but the schema/paramLabel/javadoc said only `ERROR | WARNING`, so a cold agent couldn't discover INFO.
- **Rule/computation placement is now self-describing and corrective.** `schema rule add` / `computation add` state the error-field **placement law** (a per-row rule's error field must lie inside the repeatable group it iterates; for a whole-document rule, aggregate the per-row operands), and the kernel's `MVK_ERROR_FIELD_NOT_IN_RULEGROUP` now carries a corrective `fix` instead of a bare message — across `field move` / `rule add` / `computation add` / `apply` / `batch`.
- **`diagnostics <code>` resolves the structural kernel codes** (`MVK_ERROR_FIELD_NOT_IN_RULEGROUP`, `MVK_NEG_CONDITION_IN_ITERATION`) with a meaning + fix, instead of dead-ending — an agent that meets one in an error can look it up.

## [0.4.0] — kernel 30.8.1 (A12 Tools 2025.06-ext5)

### Added

- **`apply` now covers the full delete op-set — `rule remove` and `group remove` with `cascade`.** A "delete a rule" or "delete a referenced group and its referrers" edit could not be expressed inside an `apply` transaction (`rule remove` was an unknown op; `group remove` rejected `cascade` as an unknown arg), forcing a drop to standalone calls mid-transaction. `apply` now has a `rule remove` op, and its `group remove` op takes the same `cascade` safe-delete gate as the standalone verb — both single-sourced with the standalone verbs so they can't drift.
- **`operators <substring>` now filters the catalog.** Narrowing the operator list was only possible via `--kind`/`--keyword`; an agent's natural reach — a bare `operators date` — failed as "no such operator." A non-id positional is now a substring filter (matched over id, keyword, and meaning) that composes with `--kind`/`--keyword`; an exact id still returns that operator's full semantics.

### Changed

- **`workspace list`/`graph`/`validate`/`roles` now honor `-w/--workspace` (and `$DMTOOL_WORKSPACE`), not only a positional `<dir>`.** Having set the workspace root once with `-w`, an agent's `dmtool -w <dir> workspace list` used to fail with `Missing required parameter: '<dir>'` because these verbs took the directory *only* positionally and ignored the inherited `-w`. The directory now falls back to `-w` (then `$DMTOOL_WORKSPACE`) when the positional is omitted — the same universal option that resolves a `-m` model's includes. The positional `<dir>` still works (it's the natural scan target); this just lets the session-wide root drive `workspace` too.
- **`export` now takes the model via the universal `-m`, like every other verb.** It was the one verb that required the model as a positional argument (`dmtool export <model> [artifact]`) and silently ignored the inherited `-m`, so an agent that had learned `-m` everywhere couldn't drive it (and `export rules` mis-read "rules" as the model path). It is now `dmtool -m <model> export [<artifact>]` — one consistent shape across the whole tool; the artifact stays positional (`fields`/`groups`/`rules`/`card`, default `card`).
- **`read` is now uniform across every target: no argument lists, an argument reads one.** `rule`/`computation`/`field`/`group read` with no path now return a lean name-index (the full names + a count) — the handle to then read one — instead of erroring; `include`/`typedef read` gained the inverse (read one by alias/id). So the whole read family obeys one rule — *no argument = everything in scope; an argument = that one* — closing the dead-end where listing a kind's members wasn't discoverable. (Several served `schema <target> <op>` contracts were corrected to match the live tool — `rule read`/`rule explain` still described removed `--rule`/`--all`/`--computation` flags; `model describe` had a stale enum-values key plus a missing `locales`; and `model validate`/`rule check` falsely claimed a `data` payload when their verdict actually rides the envelope's `valid`+`diagnostics`. New guards now keep a served schema from naming a flag the command lacks or describing output the verb doesn't produce.)
- **`field modify` can now change a field's metadata in place, not just re-type it.** It is a **partial** edit: a `kind`/per-kind config re-types the field (as before), and any metadata key — requiredness, label, helper text, descriptions, annotations, the global/transient flags — changes that metadata, with no delete-and-recreate. A metadata-only edit (no `kind`) is valid, and the result's `changed.aspects` reports exactly what was touched.

### Fixed

- **`group extract` no longer produces a sub-model that's broken on its own.** When the extracted group held a rule whose condition used absolute paths (e.g. `…(/Order/Shipping/Zip)`), extract copied them verbatim into the new sub-model — whose root is now `/Shipping`, not `/Order/Shipping` — so the sub-model failed the kernel check *standalone* (the composed base+mount still validated, masking it; the break only surfaced when the sub-model was reused or edited alone). Extract now re-bases a carried rule/computation's absolute operands to the sub-model root, and validates the sub-model on its own before writing — refusing rather than emitting a broken artifact.
- **A verb that loads a model which is invalid on disk no longer dumps a raw Java stack trace.** If a stored model fails the kernel's consistency check at load (e.g. a rule referencing a root the model doesn't have), strict-loading verbs like `rule modify` used to surface an internal `ModelConsistencyException` stack trace instead of a usable error. They now return the same structured `rejected` envelope a write-time kernel rejection produces — the `MVK_*` code and message in `diagnostics[]` — at one seam covering every verb. (`model validate` was already tolerant and is unchanged.)
- **`rule read` and `model describe` now give that same structured rejection on an invalid-on-disk model, not a plain `"… failed:"` string.** They caught the load exception internally and reported a bare message; they now re-throw it so the one seam renders the full `rejected` envelope (`MVK_*` code in `diagnostics[]`), consistent with every other load verb.
- **Adding a field or group *into an include-mount group* is now refused, not silently destructive.** A group that mounts an include (e.g. `/Person/Addr` mounting an `address` model) carries its content *from* that include. Adding a local field/group into it used to report `applied` while the kernel then quietly stopped expanding the include — the mounted fields *vanished* from the model, yet it still validated. It's now refused up front with `RK_MOUNT_NOT_EDITABLE` ("…would shadow the mounted content"), pointing you to edit the included model or pick a non-mount group; nothing is written. The same guard backstops the `apply`/`batch` ops.
- **A model created with a non-English display locale (e.g. `model new --locale de_DE`) now still accepts English rule conditions.** The new model derived its condition *keyword* language from its display locale, so a German model demanded German keywords (`FeldAngegeben`/`Und`) and rejected the English DSL (`FieldFilled`/`And`) shown everywhere — the skill, `operators`, the renderer — with a raw `MVK_UNEXPECTED_TOKEN`. Condition language is independent of display locale (the kernel's own German models author English-keyword rules), so a fresh model now defaults to English keywords whatever its locale — for rule conditions **and** computation operations/preconditions alike (it is one model-level setting); the locale governs labels and messages only. The rare German-keyword model is an explicit `config modify --condition-language de_DE`. (`model new --locale`'s help no longer mis-describes itself as setting "the condition + default locale".)
- **`group extract` now names the new sub-model after its reference, not the parent model.** Extracting a group into a sub-model copied the parent's model name verbatim, so an `address` sub-model extracted from a `person` model stayed internally named `person`. The sub-model's `modelInfo.name` is now set to the `--reference` id.
- **`rule add` / `computation add` no longer crash with a raw Java stack trace when a name is already taken.** Re-adding a rule/computation whose code is already used (or targeting a group that doesn't exist) used to let an internal kernel exception escape as an unreadable stack trace. It's now a clean, structured refusal — `RK_NAME_EXISTS` (with a did-this-instead fix) or `RK_NO_SUCH_GROUP` — and nothing is written. The same hardening covers every verb that shares the validation path (`rule check`, `patterns`, `model eval`), so none of them can leak a stack trace on a malformed candidate.
- **A typo'd or out-of-scope key in a JSON spec is now rejected, not silently ignored.** `field add`/`field modify`, `rule add`, `computation add`, and the `apply`/`batch` ops validate their spec against the published schema — **top-level and inside nested config blocks** — so an unrecognized key (a misspelled `requried`, a `string.maxLenght`) is refused with a corrective `RK_UNKNOWN_ARG` (and a did-you-mean) instead of being parsed-and-dropped into a false `applied`.
- **A recognized spec key carrying a *wrong-shaped value* is now rejected, not silently swallowed.** The check above guards key *names*; this guards their *values* — schema-driven, across **every** spec verb (`field add`/`field modify`, `rule add`, `computation add`, and `apply`/`batch`). A `required:"MANDATORY"` (the kernel's own term — silently became the OPTIONAL default), an `annotations` *array* (the schema declares an object `{ "name": "value" }`), a localized-text field (`label`/`helperText`/`internalDescription`/`externalDescription`) given as a bare string, or a rule's `messages` given as a string instead of an array — each used to be coerced to a default/empty while still reporting `applied`. Each is now refused with a structured **`RK_INVALID_VALUE`** that names the key and the type the schema expects. Locked by an "applied ⇒ effected" round-trip guard.
- **`field modify` no longer silently drops the metadata you give it.** Passing a `label`/`helperText`/`required` to a re-type used to report `applied` while changing nothing (a false success); those aspects are now applied, and a spec that would change nothing is refused instead of reported as applied. The same fix covers the `apply`/`batch` field-modify op.
- **An upgraded plugin now actually picks up the new binary.** The downloader cached `dmtool` at a version-less path and only fetched it when absent, so once any binary was cached, bumping the plugin never replaced it — an upgrade kept running the old binary (and the v0.3.2 download-pin fix couldn't reach anyone who already had one cached). The cache is now **keyed by version**: a newer plugin re-downloads the matching binary on its first session, prunes the old one, and keeps the deterministic `…/bin/dmtool` path as a symlink to the current version. (Already running a stale binary from before this fix? Upgrading to this version self-heals on the next session.)
